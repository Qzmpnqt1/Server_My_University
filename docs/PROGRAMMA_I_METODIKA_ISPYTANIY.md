# Программа и методика испытаний ПО «Мой ВУЗ»

## 1. Цель

Проверить соответствие реализованного клиент-серверного комплекса требованиям технического задания: роли, регистрация, расписание, успеваемость, чаты, безопасность, устойчивость основных сценариев.

## 2. Объект испытаний

- Backend: `Server_My_University` (Spring Boot, MySQL, Cassandra).  
- Клиент: `App_My_University` (Android).  
- Контур: HTTPS API, Docker Compose (опционально).

## 5. Ручные сценарии (минимум)

### 5.1 Регистрация и роли

1. Гость подаёт заявку (Android).  
2. Администратор видит заявку, одобряет / отклоняет.  
3. После одобрения — вход, получение JWT, доступ к защищённым разделам.  
**Ожидание:** отклонённая заявка не создаёт активную учётную запись.

### 5.2 Расписание

1. Администратор создаёт занятие.  
2. Попытка создать пересечение по преподавателю / группе / аудитории.  
**Ожидание:** отказ с сообщением о конфликте.  
3. Студент видит расписание группы; преподаватель — своё.

### 5.3 Успеваемость

1. Преподаватель выставляет итог и практики по **своей** дисциплине и **согласованному курсу** группы.  
2. Студент видит зачётную книжку без дисциплин чужого курса.  
**Ожидание:** итоговая оценка не выводится автоматически из практик.

### 5.4 Сообщения

1. Отправка сообщения студент ↔ преподаватель / администратор.  
2. Повторный вход — история подгружается.  
**Ожидание:** данные из Cassandra доступны после перезапуска сервера (при сохранённом volume).

### 5.5 Уведомления

1. После изменения оценки / расписания / сообщения у соответствующего пользователя появляется запись в **Профиль → Уведомления**.

## 6. Критерии приёмки

- Все сценарии раздела 5 выполняются без критических дефектов.  
- Автотесты A1–A4 завершаются успешно на эталонной среде.  
- Нет утечки данных между ролями при прямых вызовах API с чужим токеном (выборочная проверка).
